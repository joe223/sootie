# Verification Matrix

This matrix tracks the current implementation against the active Sootie
completion criteria. It is intentionally evidence-based: a green test is useful
only when it covers the requirement it is cited for.

## Success Criteria

1. Sootie is implemented as an independent Rust repository.
2. The public MCP surface uses Sootie tool names while keeping compatible data
   and argument shapes.
3. The feature surface includes desktop perception, actions, waits, recipes,
   visual grounding, annotation, learning, and browser automation.
4. The implementation supports macOS, Linux, and Windows backends.
5. Browser targets support CDP when remote debugging is enabled; CDP is the
   only intentional capability beyond the selected public tool contract.
6. macOS runtime behavior is verified through the real MCP stdio path.
7. Linux and Windows code paths compile and lint cleanly; real desktop runtime
   tests remain external to this machine.

## Current Audit Result

As of 2026-05-17, the active acceptance scope has current evidence. The Rust
implementation, public tool contract, local macOS runtime path, release MCP
stdio path, workspace tests, package contents, documentation placement, and
client configuration gates have current evidence. A production panic audit also
found no Rust `unwrap`, `expect`, `panic`, `todo`, or `unimplemented` use outside
test modules; the only remaining text hit is `ObjC.unwrap` inside a macOS JXA
script string. The current acceptance scope requires full macOS runtime
evidence, while Linux and Windows require implementation plus build/link
evidence. The final aggregate verifier passes with macOS runtime evidence at
`/private/tmp/sootie-macos-runtime-evidence2/macos-evidence.json`, Linux
build-only evidence at `/private/tmp/sootie-linux-build-evidence5/linux-evidence.json`,
and Windows build-only evidence at
`/private/tmp/sootie-windows-build-evidence2/windows-evidence.json`.

The third-party client matrix still keeps Claude Code fresh-session tool-call
evidence as a separate external-client follow-up because local Claude CLI
session/auth/network startup blocked before a Sootie tool invocation. That gap
does not change the core Sootie runtime/build acceptance result.

`node docs/development/verify-runtime-evidence.mjs` is the final evidence
aggregator. For the current acceptance scope it must pass with macOS in runtime
mode and Linux/Windows in build-only mode:

```bash
node docs/development/verify-runtime-evidence.mjs \
  --evidence path/to/macos-evidence.json \
  --evidence path/to/linux-evidence.json \
  --evidence path/to/windows-evidence.json \
  --build-only linux \
  --build-only windows
```

A one-platform `--platform` run is only a partial audit.
